Q01 What is Flutter and why would you choose it?
Easy▼
Answer

Flutter is Google's open-source UI toolkit for building natively compiled applications for mobile, web, and desktop from a single codebase using the Dart language.

Key reasons to choose Flutter:

  • Single codebase — one codebase targets iOS, Android, Web, macOS, Windows, Linux
  • Hot Reload — see UI changes instantly without restarting the app or losing state
  • High performance — Dart compiles directly to native ARM code; no JavaScript bridge
  • Rich widget library — Material and Cupertino widgets for platform-specific look & feel
  • Great for MVPs — faster time-to-market with native-like performance
Flutter's rendering engine (Skia/Impeller) draws every pixel itself, guaranteeing 60/120fps consistent rendering across platforms.
Q02 What is the difference between StatelessWidget and StatefulWidget?
Easy▼
Answer
  • StatelessWidget — immutable; its properties cannot change after creation. Use for static UI like Text, Icon, or any widget that doesn't need to update (e.g., labels, icons).
  • StatefulWidget — mutable; maintains a State object that can be rebuilt via setState(). Use when the widget needs to respond to user interaction, timer events, or API responses.
// Stateless — never changes
class MyLabel extends StatelessWidget {
  Widget build(BuildContext context) => Text('Hello');
}

// Stateful — can change
class Counter extends StatefulWidget {
  _CounterState createState() => _CounterState();
}
class _CounterState extends State<Counter> {
  int count = 0;
  Widget build(context) => Text('$count');
}
Q03 What is BuildContext in Flutter?
Easy▼
Answer

BuildContext is a reference to the location of a widget within the widget tree. It provides access to:

  • The widget's position in the tree (used by Navigator, Theme, MediaQuery)
  • Ancestor widgets via context.read<T>() or Provider.of<T>(context)
  • Screen size via MediaQuery.of(context).size
Every widget's build(BuildContext context) method receives a fresh context. Never use a context after its widget has been disposed.
Q04 What is Hot Reload vs Hot Restart?
Easy▼
Answer
  • Hot Reload — Injects updated code into the Dart VM and rebuilds the widget tree. Preserves app state. Does NOT re-run main() or initState(). Best for UI tweaks.
  • Hot Restart — Recompiles the full app and restarts from scratch. Loses app state. Useful after changing app logic, adding new dependencies, or modifying initState().
Q05 What is the StatefulWidget lifecycle?
Easy▼
Answer
  1. createState() — Creates the State object. Called once.
  2. initState() — First method called on State. Initialize variables, start animations, subscribe to streams.
  3. didChangeDependencies() — Called after initState and whenever an InheritedWidget dependency changes.
  4. build() — Returns the widget tree. Called every time state changes.
  5. didUpdateWidget() — Called when parent widget passes new configuration.
  6. setState() — Notifies framework to rebuild this widget.
  7. deactivate() — Called when widget removed from tree temporarily.
  8. dispose() — Final cleanup. Cancel timers, close streams, release controllers here.
Q06 What is the difference between SizedBox and Container?
Easy▼
Answer
  • SizedBox — Lightweight widget for fixed width/height or spacing. No decoration, no padding, no color. Use when you just need size or an empty gap.
  • Container — Feature-rich box that supports padding, margin, color, decoration, transform, constraints, and child alignment. Heavier than SizedBox.
Prefer SizedBox for spacing between widgets instead of Container for better performance.
Q07 What is the difference between Expanded and Flexible?
Easy▼
Answer
  • Expanded — Forces its child to fill all available space in the main axis. It's equivalent to Flexible(fit: FlexFit.tight).
  • Flexible — Allows its child to fill available space but the child can be smaller (FlexFit.loose by default — WRAP_CONTENT).

Both accept a flex factor to distribute space proportionally among siblings.

Q08 What are final, const, and static in Dart?
Easy▼
Answer
  • const — Compile-time constant. Value must be known at compile time. Creates a canonical instance (memory efficient). const a = 1;
  • final — Runtime constant. Value assigned once at runtime, never changes after. final name = getName();
  • static — Class-level variable/method shared across all instances. Accessed via the class name, not an instance.
Q09 What is pubspec.yaml?
Easy▼
Answer

The project configuration file written in YAML. It defines:

  • dependencies — packages required in production (e.g., dio, bloc)
  • dev_dependencies — packages only for development/testing (e.g., mockito, flutter_test)
  • Flutter SDK constraints, assets (images, fonts), and project metadata (name, version)
Q10 What is the difference between ListView and ListView.builder?
Easy▼
Answer
  • ListView — Creates all children at once (eager loading). Good for small, fixed lists that all fit on screen.
  • ListView.builder — Creates items lazily (on demand) as they scroll into view. Essential for large or infinite lists. Much better performance.
ListView.builder(
  itemCount: items.length,
  itemBuilder: (context, index) => ListTile(title: Text(items[index])),
)

Q11 Explain Future, Stream, async, and await in Dart.
Medium▼
Answer
  • Future<T> — Represents a single async value that will be available later. Like a Promise in JS. Completes once (success or error).
  • Stream<T> — Represents a sequence of async events over time (multiple values). Like an Observable. Used for real-time data (websockets, location, etc.).
  • async — Marks a function as asynchronous; it returns a Future implicitly.
  • await — Pauses execution inside an async function until the Future completes.
  • async* — Used to create a Stream (generator function). Used with yield.
  • yield — Emits a value into a Stream (used inside async* functions).
// Future example
Future<String> fetchName() async {
  await Future.delayed(Duration(seconds: 1));
  return 'Flutter';
}

// Stream example (async*)
Stream<int> counter() async* {
  for (int i = 0; i < 5; i++) {
    await Future.delayed(Duration(seconds: 1));
    yield i;
  }
}
Q12 What is the difference between FutureBuilder and StreamBuilder?
Medium▼
Answer
  • FutureBuilder — Builds UI from a single Future. Use for one-time async operations: HTTP requests, reading a file, getting device battery. Rebuilds once when the Future completes.
  • StreamBuilder — Rebuilds UI every time the Stream emits a new event. Use for continuous data: location updates, real-time chat, music playback progress.
Both handle ConnectionState (none, waiting, active, done) and snapshot data/error in their builder callbacks.
Q13 Single-subscription vs Broadcast streams — what's the difference?
Medium▼
Answer
  • Single-subscription — Can only have one listener at a time. Default stream type. If you listen again, it throws an error. Good for file I/O, HTTP responses.
  • Broadcast — Multiple listeners allowed simultaneously. Events are fired whether or not there are listeners. Good for UI event streams, state streams (BLoC uses this).
// Convert to broadcast
final broadcastStream = myStream.asBroadcastStream();
Q14 What are Keys in Flutter and when do you use them?
Medium▼
Answer

Keys help Flutter uniquely identify widgets in the widget tree, preserving state when widgets move or are rebuilt. Types:

  • ValueKey — Identity based on a value (e.g., ValueKey('user-42'))
  • ObjectKey — Identity based on an object reference
  • UniqueKey — Always unique; forces widget recreation (use sparingly)
  • GlobalKey — Access widget state from anywhere in the tree; can also get render box info
  • PageStorageKey — Preserve scroll position across navigations
Use keys when reordering StatefulWidgets in a list — without keys, Flutter might associate the wrong state to the wrong widget.
Q15 What is the BLoC pattern and how does it work?
Medium▼
Answer

BLoC (Business Logic Component) separates UI from business logic using Events and States:

  • Event — User action dispatched from UI (e.g., LoginPressed)
  • Bloc — Receives events, processes them, emits new states via an EventHandler
  • State — Represents the UI at a point in time (e.g., LoginLoading, LoginSuccess)

Key BLoC widgets: BlocProvider (creates bloc), BlocBuilder (rebuilds on state), BlocListener (side effects only), BlocConsumer (both), BlocSelector (filter specific state slices).

// Cubit (simpler BLoC)
class CounterCubit extends Cubit<int> {
  CounterCubit() : super(0);
  void increment() => emit(state + 1);
}
Q16 What is buildWhen and listenWhen in BLoC?
Medium▼
Answer
  • buildWhen(prev, curr) — A condition in BlocBuilder. If it returns false, the widget is NOT rebuilt for that state change. Prevents unnecessary UI rebuilds.
  • listenWhen(prev, curr) — Same concept for BlocListener. If false, the listener callback is not called.
BlocBuilder<CounterBloc, int>(
  buildWhen: (prev, curr) => curr % 2 == 0, // rebuild only on even numbers
  builder: (context, state) => Text('$state'),
)
Q17 Provider vs BLoC — when to use which?
Medium▼
Answer
  • Provider — Simpler, less boilerplate. Best for small/medium apps, simple state, or when you need scoped state without complex event flows.
  • BLoC — Strict separation of concerns with Events/States. Best for large apps, complex async flows, when testability is critical, or when multiple developers are working on the same feature.
BLoC's explicit event/state model makes it easier to trace bugs and write unit tests. Provider is faster to set up but harder to scale.
Q18 What is a Mixin in Dart?
Medium▼
Answer

A mixin is a way to reuse code across multiple class hierarchies without using inheritance. It "plugs in" methods and properties into a class using the with keyword.

mixin LogMixin {
  void log(String msg) => print('[LOG] $msg');
}

class ApiService with LogMixin {
  void fetchData() {
    log('Fetching...'); // from mixin
  }
}
SingleTickerProviderStateMixin is a famous Flutter mixin used for animation controllers.
Q19 What is InheritedWidget?
Medium▼
Answer

InheritedWidget is Flutter's built-in mechanism for passing data down the widget tree efficiently without prop drilling. Descendant widgets can access the nearest ancestor InheritedWidget using context.dependOnInheritedWidgetOfExactType().

When the InheritedWidget changes, only widgets that depend on it are rebuilt. Provider, Theme, MediaQuery, and Navigator are all built on top of InheritedWidget.

Q20 What is Equatable and why is it used in BLoC?
Medium▼
Answer

By default, Dart compares objects by reference (== checks if they're the same instance). BLoC only emits a new state if it's different from the previous one.

Equatable overrides == and hashCode based on props, so two objects with the same data are considered equal, preventing unnecessary state emissions and rebuilds.

class UserState extends Equatable {
  final String name;
  const UserState(this.name);
  @override
  List<Object> get props => [name];
}
Q21 What are the SOLID principles?
Medium▼
Answer
  • S — Single Responsibility: A class should do one thing. One reason to change.
  • O — Open/Closed: Open for extension, closed for modification. Add behavior via inheritance/composition, not editing existing code.
  • L — Liskov Substitution: Subclasses should be substitutable for their base class without breaking the program.
  • I — Interface Segregation: Many small, specific interfaces are better than one large general-purpose interface. Don't force classes to implement methods they don't need.
  • D — Dependency Inversion: High-level modules should depend on abstractions, not concretions. Both should depend on interfaces.
Q22 How does Flutter handle routing and navigation?
Medium▼
Answer
  • Navigator 1.0 — Imperative. Navigator.push(), Navigator.pop(). Simple but hard to handle deep links and web URLs.
  • Navigator 2.0 — Declarative. Full control over the navigation stack via a Router and RouteInformationParser. Complex but powerful.
  • GoRouter — The recommended package. Declarative URL-based routing with deep linking, guards, and named routes. Simple API over Navigator 2.0.
  • auto_route — Code-generated routing with type-safe arguments.
Q23 What is Dependency Injection in Flutter?
Medium▼
Answer

DI is a design pattern where a class receives its dependencies from outside rather than creating them itself. This decouples components making code more testable and maintainable.

In Flutter, common DI approaches:

  • Constructor injection — Pass dependencies via constructor
  • get_it — Service locator pattern; register singletons/factories globally
  • Provider / Riverpod — DI via the widget tree
  • injectable — Code generation for get_it
Q24 What is the difference between List, Map, and Set in Dart?
Medium▼
Answer
  • List — Ordered collection, allows duplicates. Index-based access. [1, 2, 3]
  • Set — Unordered collection, no duplicates. Faster lookup than List. {1, 2, 3}
  • Map — Key-value pairs. Keys are unique. {'name': 'Ali', 'age': 25}
Use Set when you need to check for containment frequently — set.contains(x) is O(1) vs O(n) for List.
Q25 What is Deep Linking in Flutter?
Medium▼
Answer

Deep linking allows a URL to open a specific screen inside your app directly. Components of a deep link URL:

  • Scheme — Protocol (https:// or custom myapp://)
  • Host — Domain (example.com)
  • Path — Route within app (/profile/42)

Flutter supports deep linking via GoRouter's initialLocation or by configuring AndroidManifest.xml (Android) and Info.plist (iOS). Dynamic linking (Firebase) also handles cases where the app isn't installed yet.

Q26 What are JIT and AOT compilation? Why does Flutter use both?
Medium▼
Answer
  • JIT (Just-In-Time) — Used in debug mode. Compiles Dart code at runtime. Enables hot reload and faster development iterations. Larger app size, slower startup.
  • AOT (Ahead-Of-Time) — Used in release mode. Compiles Dart to native ARM code before the app runs. Results in faster startup, better performance, smaller runtime footprint.
This dual approach gives developers the best of both worlds: fast development with JIT, and production performance with AOT.
Q27 What are the three trees in Flutter (Widget, Element, RenderObject)?
Medium▼
Answer
  • Widget Tree — Blueprint/configuration. Immutable. Describes what the UI should look like. Rebuilt frequently, but cheap because widgets are just config objects.
  • Element Tree — Represents the actual instantiation of each widget in the tree. Persists across rebuilds, tracking widget lifecycle (Stateless or Stateful element).
  • RenderObject Tree — Handles actual layout, painting, and hit testing. The most expensive tree — Flutter avoids rebuilding it unless necessary.
The Element tree is Flutter's reconciliation layer — it matches the new Widget tree against the old one to minimize RenderObject updates.
Q28 What is Typedef in Dart?
Medium▼
Answer

typedef creates an alias for a function type, making it more readable and reusable.

typedef IntCallback = void Function(int value);

void doSomething(IntCallback callback) {
  callback(42);
}

// Usage
doSomething((val) => print(val)); // prints 42

Also used for generic type aliases with Dart generics: typedef JsonMap = Map<String, dynamic>;

Q29 How do you handle errors in networking / API calls in Flutter?
Medium▼
Answer
  • Wrap API calls in try/catch to handle DioException, SocketException, TimeoutException
  • Check response.statusCode for HTTP errors (4xx, 5xx)
  • Use Interceptors (in Dio) to centrally handle auth errors, retry logic, logging
  • Return a sealed class or Either<Failure, Success> (from dartz or fpdart) from repositories
  • Emit error states from BLoC to show error UI
Q30 What is Unit Testing and Mocking in Flutter?
Medium▼
Answer

Flutter has 3 testing layers:

  • Unit tests — Test a single function/class in isolation. Fast. No Flutter framework needed.
  • Widget tests — Test widgets by rendering them in a test environment. WidgetTester simulates interactions.
  • Integration tests — Full app tests on a real device/emulator. Slowest but most realistic.

Mock — Simulates a dependency and verifies interactions (expects certain methods to be called). Use mockito or mocktail.
Stub — Provides a fixed response without verifying how it was called.

Q31 How do you optimize Flutter app performance?
Hard▼
Answer
  • Use const constructors — Prevents widget reconstruction if data hasn't changed
  • Minimize setState() scope — Call it on the smallest possible widget subtree
  • ListView.builder — Lazy render for large lists
  • RepaintBoundary — Isolates a subtree from parent repaints (useful for complex animations)
  • Avoid rebuilding expensive widgets — Use buildWhen in BLoC, select() in Provider/Riverpod
  • Image optimization — Use cached_network_image, resize images to display size
  • Isolates — Offload heavy computation off the main isolate
  • Profile mode — Use Flutter DevTools to identify jank and expensive widgets
Q32 What is Clean Architecture in Flutter?
Hard▼
Answer

Clean Architecture divides the app into 3 independent layers with strict dependency rules (inner layers don't know about outer layers):

  • Data Layer — Data sources (API, local DB), DTOs, Repository implementations
  • Domain Layer — Entities, Use Cases (business rules), Repository interfaces. Pure Dart, no Flutter/framework dependencies.
  • Presentation Layer — BLoC/Cubit, widgets, pages. Depends on Domain.

Dependencies flow inward: Presentation → Domain ← Data. The Repository interface in Domain is implemented by Data but consumed by Presentation, enforcing the Dependency Inversion Principle.

Q33 What are Platform Channels and how do you write native code?
Hard▼
Answer

Platform Channels allow Flutter (Dart) to communicate with native code (Kotlin/Java for Android, Swift/Objective-C for iOS). Uses message passing with a codec.

// Dart side
const channel = MethodChannel('com.example/battery');
final level = await channel.invokeMethod<int>('getBatteryLevel');

// Kotlin side (Android)
MethodChannel(flutterEngine.dartExecutor, "com.example/battery")
  .setMethodCallHandler { call, result ->
    if (call.method == "getBatteryLevel") result.success(getBatteryLevel())
    else result.notImplemented()
  }

Types: MethodChannel (request/response), EventChannel (streams of events), BasicMessageChannel (arbitrary messages).

Q34 How do you implement offline mode in a Flutter app?
Hard▼
Answer
  • Cache data locally — Use Hive, SQLite (sqflite), or Isar for structured local storage
  • Detect connectivity — Use connectivity_plus to listen for network changes
  • Optimistic UI — Update UI immediately, sync to server when back online
  • Sync queue — Queue failed requests locally; replay when connectivity returns
  • Data conflict resolution — Use timestamps (last-write-wins) or version vectors for conflict detection
  • Offline auth — Cache tokens locally with flutter_secure_storage
For data inconsistency between app and server, prefer a "server-wins" policy for critical data, or prompt the user to resolve conflicts manually.
Q35 How do you secure a Flutter app? (JWT, SSL, storage)
Hard▼
Answer
  • JWT security — Store in flutter_secure_storage (Keychain on iOS, Keystore on Android). Set short expiry. Use HTTPS only. Implement refresh tokens.
  • SSL Pinning — Embed the server's certificate in the app. Reject connections if certificate doesn't match, preventing MITM attacks.
  • Code obfuscation — Use --obfuscate flag in release builds to hide class/method names
  • API key protection — Never hardcode in source. Use environment variables, obfuscation, or backend proxy.
  • Jailbreak detection — Use flutter_jailbreak_detection package
  • Secure network — Always use HTTPS/TLS. Implement certificate pinning with Dio.
  • Background snapshot protection — Use secure_application to blur app when backgrounded
Q36 What are Isolates in Dart and when do you use them?
Hard▼
Answer

Dart is single-threaded. Isolates are independent workers that run code in parallel on separate threads, communicating only via message passing (no shared memory).

Use Isolates when you have CPU-intensive work that would block the main thread and cause jank:

  • Parsing large JSON responses
  • Image processing / compression
  • Complex calculations or encryption
// Simple isolate via compute()
final result = await compute(parseJsonIsolate, jsonString);

List<User> parseJsonIsolate(String json) {
  return (jsonDecode(json) as List).map((e) => User.fromJson(e)).toList();
}
Q37 What is the difference between Equatable and Freezed?
Hard▼
Answer
  • Equatable — Lightweight. Manually list props. Only provides == and hashCode. No code generation needed. Good for BLoC states/events.
  • Freezed — Code-generated. Provides: value equality, copyWith(), toString(), union types (sealed classes), JSON serialization (with json_serializable). More powerful but requires build_runner.
// Freezed example
@freezed
class User with _$User {
  const factory User({required String name, required int age}) = _User;
  factory User.fromJson(Map<String, dynamic> json) => _$UserFromJson(json);
}
Q38 What is the Flutter Architecture (Embedder, Engine, Framework)?
Hard▼
Answer
  • Embedder (Bottom) — Platform-specific code that integrates Flutter with the OS. Handles input, accessibility, rendering surfaces. Written in Java/C++ (Android), Objective-C/Swift (iOS).
  • Flutter Engine (Middle) — Written in C++. Handles Dart VM execution, graphics rendering (Skia/Impeller), file I/O, networking. Exposes dart:ui API to the framework.
  • Flutter Framework (Top) — Written in Dart. Where developers work. Layers: Foundation → Animation/Gestures → Rendering → Widgets → Material/Cupertino.
Q39 How do you implement animations in Flutter?
Hard▼
Answer
  • Implicit Animations — Automatically animate between values. Widgets like AnimatedContainer, AnimatedOpacity, AnimatedSwitcher. Simple, no controller needed.
  • Explicit Animations — Full control via AnimationController + Tween + AnimatedBuilder. Use when you need precise timing, sequences, or custom curves.
  • Hero Animation — Smooth shared-element transition between routes. Wrap both widgets in Hero with matching tag.
  • Lottie / Rive — Pre-built vector animations for complex visuals.

Q41 What is a Sealed Class in Flutter/Dart?
Hard▼
Answer

Introduced in Dart 3. A sealed class restricts its subclasses to a finite, known set defined in the same library. Enables exhaustive pattern matching with switch.

sealed class AuthState {}
class AuthLoading extends AuthState {}
class AuthSuccess extends AuthState { final User user; AuthSuccess(this.user); }
class AuthError extends AuthState { final String message; AuthError(this.message); }

// Exhaustive switch — compiler ensures all cases handled
switch (state) {
  case AuthLoading(): ...
  case AuthSuccess(): ...
  case AuthError(): ...
}
Q42 What is RepaintBoundary and when should you use it?
Hard▼
Answer

RepaintBoundary creates a separate layer for its subtree, so when that subtree repaints, it doesn't trigger repaints in parent or sibling subtrees.

Use when:

  • A subtree animates frequently (e.g., an animated progress indicator in a complex screen)
  • A child has complex painting that is independent of parent changes
  • Flutter DevTools shows unnecessary repaints (red checkerboard in paint debugging)
Overusing RepaintBoundary increases memory usage (each boundary creates a separate texture). Use it surgically based on profiling.
Q43 What is Push Notification and how do you implement it in Flutter?
Hard▼
Answer

Push notifications alert users to events when the app is in the background or closed. Implementation using Firebase Cloud Messaging (FCM):

  • Integrate firebase_messaging package
  • Request notification permissions (messaging.requestPermission())
  • Get FCM device token and send it to your backend
  • Handle messages: FirebaseMessaging.onMessage (foreground), onMessageOpenedApp (background tap), getInitialMessage() (terminated)
  • For local notifications, use flutter_local_notifications

Q44 What is Concurrency in Dart and how does the Event Loop work?
Expert▼
Answer

Dart is single-threaded and uses an Event Loop for concurrency. The event loop has two queues:

  • Microtask Queue — Highest priority. Processed before the event queue. Dart Futures that complete synchronously go here (Future.microtask, scheduleMicrotask).
  • Event Queue — Regular async events: I/O, timer callbacks, user events, Future.delayed.

The loop: process all microtasks → process one event → repeat. True parallelism requires Isolates, which have their own event loops and communicate via ports.

Q45 What is FFI in Dart?
Expert▼
Answer

FFI (Foreign Function Interface) allows Dart code to directly call native C libraries without platform channels. It's faster than method channels for performance-critical native code.

import 'dart:ffi';

final dylib = DynamicLibrary.open('libnative.so');
final nativeAdd = dylib.lookupFunction<
  Int32 Function(Int32, Int32),
  int Function(int, int)
>('add');

print(nativeAdd(3, 4)); // 7

Use FFI for: encryption libraries, image processing codecs, or integrating existing C/C++ SDKs.

Q46 What are Generics in Dart and why are they important?
Expert▼
Answer

Generics allow you to write type-safe, reusable code without sacrificing flexibility. Dart generics are reified — type information is preserved at runtime (unlike Java's type erasure).

// Generic repository
abstract class Repository<T> {
  Future<List<T>> getAll();
  Future<T> getById(int id);
}

// Generic result type
sealed class Result<T> {}
class Success<T> extends Result<T> { final T data; Success(this.data); }
class Failure<T> extends Result<T> { final String err; Failure(this.err); }
Q47 What are Runes and Symbols in Dart?
Expert▼
Answer
  • Runes — A String is a sequence of UTF-16 code units. Runes expose the Unicode code points of a string. Use string.runes to iterate actual characters including emoji and non-BMP characters. '😀'.runes.first → 128512
  • Symbol — Represents an operator or identifier in Dart programs. Created with #identifier. Used primarily in reflection via dart:mirrors or code generation. E.g., #print refers to the print identifier.
Q48 What is a Dart Callable Class?
Expert▼
Answer

A callable class implements a call() method, allowing instances to be invoked like functions. Useful for creating reusable, stateful function-like objects.

class Multiplier {
  final int factor;
  Multiplier(this.factor);
  int call(int value) => value * factor;
}

final triple = Multiplier(3);
print(triple(7)); // 21 — called like a function
Q49 What is CI/CD for Flutter apps and how do you set it up?
Expert▼
Answer

CI (Continuous Integration) — Automatically build, test, and lint on every commit/PR. Tools: GitHub Actions, Bitrise, Codemagic, CircleCI.

CD (Continuous Delivery/Deployment) — Automatically deliver signed builds to testers (Firebase App Distribution, TestFlight) or publish to app stores.

Typical Flutter CI/CD pipeline:

  1. Checkout code → flutter pub get
  2. Run flutter analyze (linting)
  3. Run flutter test (unit + widget tests)
  4. Build: flutter build apk --release / flutter build ipa
  5. Sign with keystores/certificates (stored as CI secrets)
  6. Deploy to Firebase App Distribution or Play Store/App Store
Q50 How do you handle background tasks in Flutter?
Expert▼
Answer
  • Isolates — For CPU-heavy work in parallel without blocking the UI thread
  • workmanager (package) — Schedule periodic/one-off background tasks that run even when the app is closed (uses WorkManager on Android, BGTaskScheduler on iOS)
  • flutter_background_service — Long-running foreground services (e.g., location tracking, music playback)
  • Firebase Cloud Messaging — Trigger background code via push notification data messages
iOS has strict limits on background execution. Always test background behavior on a real device, not just the simulator.
Q51 What is MVVM architecture in Flutter?
Expert▼
Answer
  • Model — Data layer: entities, repositories, API services
  • ViewModel — Holds UI state, exposes streams/ValueNotifiers, contains business logic. Implemented with ChangeNotifier, Riverpod Notifier, or Cubit.
  • View — Flutter widgets. Observes ViewModel, calls its methods on user actions. Has no business logic.

MVVM is naturally implemented in Flutter with Provider + ChangeNotifier or Riverpod. BLoC can also be considered MVVM where the Bloc is the ViewModel.

Q52 Sort an array without predefined sort methods (coding question)
Expert▼
Answer — Bubble Sort in Dart
List<int> bubbleSort(List<int> arr) {
  int n = arr.length;
  for (int i = 0; i < n - 1; i++) {
    for (int j = 0; j < n - i - 1; j++) {
      if (arr[j] > arr[j + 1]) {
        int temp = arr[j];
        arr[j] = arr[j + 1];
        arr[j + 1] = temp;
      }
    }
  }
  return arr;
}
// Time: O(n²) | Space: O(1)
// For interviews, also mention: Merge Sort O(n log n)
Q53 What are the drawbacks of Flutter?

Q54 What is tree shaking in Flutter?
Expert▼
Answer

Tree shaking is a dead code elimination technique performed during the AOT build process. The Dart compiler analyzes which code is actually reachable from your app's entry point and removes all unreachable code from the final binary.

This is why kDebugMode/kReleaseMode constant flags work: code inside if (kReleaseMode) blocks is tree-shaken away in debug builds (and vice versa), with zero runtime cost.

Q55 What is GraphQL and how does it differ from REST?
Expert▼
Answer
  • REST — Multiple endpoints, each returning a fixed data shape. Can suffer from over-fetching (too much data) or under-fetching (multiple requests needed).
  • GraphQL — Single endpoint. Client specifies exactly what fields it needs in a query. Eliminates over/under-fetching. Strongly typed schema. Supports real-time with Subscriptions.

In Flutter, use the graphql_flutter package. Queries, Mutations, and Subscriptions are defined as strings and executed via the GraphQLClient.
